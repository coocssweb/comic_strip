# 编写实施计划

## 概述

编写全面的实施计划，假设工程师对我们的代码库零上下文、且审美品味存疑。记录他们需要知道的一切：每个任务要改哪些文件、代码、测试、可能需要查阅的文档、如何验证。把整个计划拆成小粒度任务。DRY、YAGNI、TDD、频繁提交。

假设他们是熟练的开发者，但对我们的工具链和问题域几乎一无所知。假设他们不太擅长测试设计。

**启动时声明：** "我正在使用 writing-plans skill 来创建实施计划。"

**上下文：** 应在独立的工作树中运行（由 brainstorming skill 创建）。

**计划保存位置：** `docs/superpowers/plans/YYYY-MM-DD-<功能名称>.md`
- （用户对计划存放位置的偏好优先于此默认值）

## 范围检查

如果规格说明涵盖多个独立子系统，应该在头脑风暴阶段就拆分为子项目规格。如果没有拆分，建议将其拆成独立的计划——每个子系统一个。每个计划应能独立产出可运行、可测试的软件。

## 文件结构

在定义任务之前，先梳理哪些文件需要创建或修改，以及每个文件的职责。分解决策在这一步锁定。

- 设计边界清晰、接口明确的单元。每个文件应有一个清晰的职责。
- 你对能一次性放入上下文的代码推理效果最好，文件越聚焦编辑越可靠。优先选择小而聚焦的文件，而非大而全的文件。
- 一起变更的文件应放在一起。按职责拆分，而非按技术层拆分。
- 在已有代码库中，遵循既有模式。如果代码库使用大文件，不要单方面重构——但如果你要修改的文件已经臃肿，在计划中包含拆分是合理的。

此结构指导任务分解。每个任务应产出独立的、自洽的变更。

## 小粒度任务

**每一步是一个动作（2-5 分钟）：**
- "编写失败的测试" - 一步
- "运行确认测试失败" - 一步
- "实现最小代码使测试通过" - 一步
- "运行测试确认通过" - 一步
- "提交" - 一步

## 计划文档头部

**每个计划必须以此头部开始：**

```markdown
# [功能名称] 实施计划

> **给 AI 执行者：** 必须使用子技能：使用 subagent-driven-development（推荐）或 executing-plans 来逐任务实施此计划。步骤使用复选框（`- [ ]`）语法跟踪进度。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句话描述方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### 任务 N: [组件名称]

**文件：**
- 创建: `exact/path/to/file.py`
- 修改: `exact/path/to/existing.py:123-145`
- 测试: `tests/exact/path/to/test.py`

- [ ] **步骤 1: 编写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2: 运行测试确认失败**

运行: `pytest tests/path/test.py::test_name -v`
预期: 失败，报错 "function not defined"

- [ ] **步骤 3: 编写最小实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4: 运行测试确认通过**

运行: `pytest tests/path/test.py::test_name -v`
预期: 通过

- [ ] **步骤 5: 提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "功能: 添加具体功能"
```
````

## 禁止占位符

每一步必须包含工程师需要的实际内容。以下是**计划缺陷**——绝不允许出现：
- "待定"、"TODO"、"后续实现"、"补充细节"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上述内容编写测试"（没有实际测试代码）
- "类似任务 N"（重复写出代码——工程师可能不按顺序阅读任务）
- 只描述做什么但不展示怎么做的步骤（代码步骤必须有代码块）
- 引用了任何任务中未定义的类型、函数或方法

## 记住
- 始终使用精确文件路径
- 每一步都包含完整代码——如果步骤涉及代码变更，展示代码
- 精确的命令和预期输出
- DRY、YAGNI、TDD、频繁提交

## 自检

写完完整计划后，用全新的视角对照规格说明检查计划。这是你自己执行的检查清单——不是子代理调度。

**1. 规格覆盖：** 浏览规格说明的每个章节/需求。能否指出实现它的任务？列出所有缺口。

**2. 占位符扫描：** 搜索计划中的危险信号——上述"禁止占位符"章节中的任何模式。修复它们。

**3. 类型一致性：** 后续任务中使用的类型、方法签名和属性名是否与前面任务中定义的一致？任务 3 中叫 `clearLayers()` 但任务 7 中叫 `clearFullLayers()` 就是 bug。

如果发现问题，直接修复。无需重新检查——修完继续。如果发现规格需求没有对应任务，补上任务。

审阅检查项的详细标准见：`plan-document-reviewer-prompt.md`

## 执行交接

保存计划后，提供执行选择：

**"计划已完成并保存到 `docs/superpowers/plans/<文件名>.md`。两种执行方式：**

**1. 子代理驱动（推荐）** - 每个任务分派一个全新子代理，任务间审查，快速迭代

**2. 内联执行** - 在当前会话中使用 executing-plans 执行任务，批量执行带检查点

**选择哪种？"**

**如果选择子代理驱动：**
- **必须使用子技能：** subagent-driven-development
- 每个任务一个全新子代理 + 两阶段审查

**如果选择内联执行：**
- **必须使用子技能：** executing-plans
- 批量执行带审查检查点

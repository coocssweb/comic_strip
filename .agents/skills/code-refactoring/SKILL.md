# 代码重构流程

## 概述

重构必须先诊断再动手。跳过诊断直接改代码 = 盲目重构，容易引入新问题。

**核心原则：** 诊断 → 设计 → 确认 → 执行，每个阶段必须用户确认后才能进入下一阶段。

**违反流程的字面要求，就是违反重构的精神。**

## 铁律

```
没有完成架构诊断，禁止提出重构方案
没有用户确认设计，禁止开始改代码
```

## 适用场景

- 用户明确要求重构
- 组件/模块职责混乱需要拆分
- 代码耦合严重需要解耦
- 架构不合理需要调整

**不适用：**
- 简单 bug 修复（用 systematic-debugging）
- 纯样式调整
- 无明确业务需求的「优雅化」改动

## 三个阶段

必须按顺序执行，每个阶段需用户确认后才能进入下一阶段。

### 阶段一：架构诊断

以「资深前端架构师 + Code Reviewer」视角分析。

**这个阶段只做诊断，不改代码，不给方案。**

从四个维度分析：

| 维度 | 关注点 |
|------|--------|
| 设计层 | 抽象是否合理、数据流是否清晰、接口是否单一职责 |
| 职责划分 | 哪些职责混在一起、是否有上帝组件/万能函数 |
| 可测试性 | 外部依赖是否可 mock、副作用是否可隔离、纯函数比例 |
| 可扩展性 | 新增需求改动范围、隐式耦合、配置与逻辑是否分离 |

**输出要求：**
- 列表结构
- 每条问题说明「为什么这是问题」
- 不给解决方案，不写新代码

### 阶段二：重构设计

用户确认阶段一的诊断结果后，输出重构方案：

1. **模块/组件拆分设计** — 重构后的文件结构与组件划分
2. **职责说明** — 每个模块/组件的单一职责定义
3. **数据流 & 依赖关系** — 组件间的数据流向与依赖图
4. **契约边界** — 新增或变更的导出组件、hook、工具函数，以及它们需要声明的 props / 参数 / 返回值契约

**设计约束：**
- 职责单一
- 禁止引入额外复杂度
- 确保可回滚

**禁止行为：**
- 禁止跳过 Spec 直接在 vibe 模式下改代码
- 禁止把所有改动塞进一个 task
- 禁止在 task 中混入未经诊断的额外重构

### 阶段三：执行与验收

用户确认阶段二的设计后，按确认范围执行重构。

**执行要求：**
- 只实现阶段二确认过的拆分与边界，不顺手扩大重构范围
- 新增跨文件模块必须对照工程规则补齐类型契约：React 组件声明 props 契约，hook / 工具函数声明参数与返回值契约
- 如果当前子项目缺少声明契约所需依赖或类型基础，必须先补依赖、迁移到已有类型方案，或在任务中明确说明豁免原因

**验收要求：**
- 对照阶段二的“契约边界”逐项确认，没有新增无契约的导出组件、hook 或工具函数
- 执行与改动范围匹配的 lint、测试、构建或静态检查
- 最终反馈必须说明验证命令、结果，以及仍存在的历史问题或豁免项

## 危险信号 — 立即停下来

如果你发现自己在想：
- 「先改了再说，改完再诊断」
- 「这个很明显，不用分析」
- 「顺手把这个也重构了」
- 「用户没说要确认，直接改吧」
- 「诊断太慢了，直接出方案」
- 「不用建 Spec，直接在对话里改完更快」

**以上全部意味着：停下来，回到阶段一。**

## 常见借口

| 借口 | 现实 |
|------|------|
| 「问题很明显，不用诊断」 | 明显的问题背后往往有不明显的耦合 |
| 「用户催得急，直接改」 | 没诊断的重构比不重构更危险 |
| 「顺手改一下不影响」 | 顺手改 = 无法回滚 + 混入功能 PR |
| 「我经验丰富，不需要流程」 | 流程保护的是代码，不是质疑你的能力 |
| 「先改再让用户看」 | 改完再确认 = 沉没成本绑架决策 |

## 与稳定代码保护的关系

- 禁止为「优雅」而改动稳定代码
- 无明确业务需求或 Bug 修复目的，不允许重构
- 重构必须有明确的问题驱动，不是「看着不爽」

# SOLID 异味提示

## SRP（单一职责）

- 文件承载不相关的关注点（例如，一个文件中有 HTTP + DB + 领域规则）
- 内聚性低或有多个变更理由的大型类/模块
- 编排许多不相关步骤的函数
- 对系统了解过多的上帝对象
- 问："这个模块会变更的唯一理由是什么？"

## OCP（开闭原则）

- 添加新行为需要编辑许多 switch/if 块
- 功能增长需要修改核心逻辑而非扩展
- 没有用于变体的插件/策略/钩子点
- 问："我能否在不触碰现有代码的情况下添加新变体？"

## LSP（里氏替换）

- 子类检查具体类型或对基类方法抛出异常
- 重写方法削弱前置条件或加强后置条件
- 子类忽略或空操作父类行为
- 问："我能否在调用者不知情的情况下替换任何子类？"

## ISP（接口隔离）

- 接口方法很多，大多数实现者不使用
- 调用者为狭窄需求依赖宽接口
- 接口方法的空/桩实现
- 问："所有实现者都使用所有方法吗？"

## DIP（依赖倒置）

- 高层逻辑依赖具体 IO、存储或网络类型
- 硬编码实现而非抽象或注入
- 将业务逻辑耦合到基础设施的导入链
- 问："我能否在不更改业务逻辑的情况下替换实现？"

## 常见代码异味（超越 SOLID）

| 异味 | 说明 |
| --- | --- |
| 长方法 | 函数 > 30 行，多层嵌套 |
| 特性嫉妒 | 方法使用另一个类的数据比自己的多 |
| 数据泥团 | 相同参数组反复一起传递 |
| 原始类型偏执 | 使用字符串/数字而非领域类型 |
| 散弹式修改 | 一个变更需要跨多个文件编辑 |
| 发散式变更 | 一个文件因许多不相关原因而变更 |
| 死代码 | 不可达或从未调用的代码 |
| 投机性通用性 | 为假设的未来需求创建抽象 |
| 魔法数字/字符串 | 没有命名常量的硬编码值 |

## 重构启发式

1. 按职责拆分，而非按大小 —— 小文件仍可能违反 SRP
2. 仅在需要时引入抽象 —— 等待第二个用例
3. 保持重构增量 —— 在移动前隔离行为
4. 首先保留行为 —— 在重构前添加测试
5. 按意图命名 —— 如果命名困难，抽象可能有问题
6. 优先组合而非继承 —— 继承创建紧耦合
7. 使非法状态不可表示 —— 使用类型强制不变量

# 安全和可靠性检查清单

## 输入/输出安全

- XSS：不安全的 HTML 注入、`dangerouslySetInnerHTML`、未转义模板、innerHTML 赋值
- 注入：通过字符串拼接或模板字面量的 SQL/NoSQL/命令/GraphQL 注入
- SSRF：用户控制的 URL 在无白名单验证的情况下访问内部服务
- 路径遍历：文件路径中的用户输入未经清理（`../` 攻击）
- 原型污染：JavaScript 中不安全的对象合并（`Object.assign`、带用户输入的展开）

## AuthN/AuthZ

- 读/写操作缺少租户或所有权检查
- 新端点无认证守卫或 RBAC 执行
- 信任客户端提供的角色/标志/ID
- 破坏的访问控制（IDOR - 不安全的直接对象引用）
- 会话固定或弱会话管理

## JWT 和令牌安全

- 算法混淆攻击（接受 `none` 或期望 `RS256` 时接受 `HS256`）
- 弱密钥或硬编码密钥
- 缺少过期时间（`exp`）或未验证
- JWT 载荷中的敏感数据（令牌是 base64，非加密）
- 未验证 `iss`（签发者）或 `aud`（受众）

## 秘密和 PII

- 代码/配置/日志中的 API 密钥、令牌或凭据
- git 历史中的秘密或暴露给客户端的环境变量
- 过度记录 PII 或敏感载荷
- 错误消息中缺少数据脱敏

## 供应链和依赖

- 未固定依赖允许恶意更新
- 依赖混淆（私有包名冲突）
- 从不受信任来源或无完整性检查的 CDN 导入
- 有已知 CVE 的过时依赖

## CORS 和头部

- 过于宽松的 CORS（`Access-Control-Allow-Origin: *` 携带凭据）
- 缺少安全头部（CSP、X-Frame-Options、X-Content-Type-Options）
- 暴露内部头部或堆栈跟踪

## 运行时风险

- 无界循环、递归调用或大型内存缓冲区
- 外部调用缺少超时、重试或速率限制
- 请求路径上的阻塞操作（异步上下文中的同步 I/O）
- 资源耗尽（文件句柄、连接、内存）
- ReDoS（正则表达式拒绝服务）

## 密码学

- 弱算法（用于安全目的的 MD5、SHA1）
- 硬编码 IV 或盐
- 使用无认证的加密（ECB 模式、无 HMAC）
- 密钥长度不足

## 竞态条件

竞态条件是导致间歇性故障和安全漏洞的微妙错误。特别关注：

### 共享状态访问

- 多线程/goroutine/异步任务在无同步的情况下访问共享变量
- 并发修改的全局状态或单例
- 无适当锁定的延迟初始化（双重检查锁定问题）
- 并发上下文中使用的非线程安全集合

### 检查后操作（TOCTOU）

- 无原子操作的 `if (exists) then use` 模式
- `if (authorized) then perform` 授权可能变更
- 文件存在检查后跟文件操作
- 余额检查后跟扣款（金融操作）
- 库存检查后跟下单

### 数据库并发

- 缺少乐观锁（`version` 列、`updated_at` 检查）
- 缺少悲观锁（`SELECT FOR UPDATE`）
- 无事务隔离的读-修改-写
- 无原子操作的计数器增量（`UPDATE SET count = count + 1`）
- 并发插入中的唯一约束违规

### 分布式系统

- 共享资源缺少分布式锁
- 领导选举竞态条件
- 缓存失效竞态（写入后陈旧读取）
- 无适当排序的事件排序依赖
- 集群操作中的脑裂场景

### 要标记的常见模式

```
# 危险模式：
if not exists(key):       # TOCTOU
    create(key)

value = get(key)          # 读-修改-写
value += 1
set(key, value)

if user.balance >= amount:  # 检查后操作
    user.balance -= amount
```

### 要问的问题

- "如果两个请求同时命中此代码会发生什么？"
- "此操作是原子的还是可以被中断？"
- "此代码访问什么共享状态？"
- "在高并发下此代码如何表现？"

## 数据完整性

- 缺少事务、部分写入或不一致状态更新
- 持久化前弱验证（类型强制问题）
- 可重试操作缺少幂等性
- 并发修改导致的丢失更新

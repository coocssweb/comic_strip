# 代码审查专家

## 概述

对当前模块进行结构化审查，重点关注 SOLID、架构、移除候选和安全风险。默认仅输出审查结果，除非用户要求实施变更。

## 严重级别

| 级别 | 说明 | 示例 | 处理方式 |
| --- | --- | --- | --- |
| P0 | 关键 | 安全漏洞、数据丢失风险、正确性错误 | 必须阻止合并 |
| P1 | 高 | 逻辑错误、严重 SOLID 违规、性能退化 | 应在合并前修复 |
| P2 | 中 | 代码异味、可维护性问题、轻微 SOLID 违规 | 在此 PR 中修复或创建后续任务 |
| P3 | 低 | 样式、命名、轻微建议 | 可选改进 |

## 工作流程

### 1) 预检上下文

- 使用 `git status -sb`、`git diff --stat` 和 `git diff` 确定变更范围
- 如需要，使用 `rg` 或 `grep` 查找相关模块、用法和约定
- 识别入口点、所有权边界和关键路径（认证、支付、数据写入、网络）

边缘情况：
- 无变更：如果 `git diff` 为空，通知用户并询问是否要审查暂存变更或特定提交范围
- 大型差异（>500 行）：先按文件摘要，然后按模块/功能区域分批审查
- 混合关注点：按逻辑功能分组发现，而不仅仅是文件顺序

### 2) SOLID + 架构异味

- 加载 `references/solid-checklist.md` 获取具体提示
- 查找：
  - SRP：承载不相关职责的过载模块
  - OCP：频繁编辑以添加行为而非扩展点
  - LSP：破坏期望或需要类型检查的子类
  - ISP：包含未使用方法的宽接口
  - DIP：高层逻辑绑定到低层实现
- 提出重构时，解释为什么它能改善内聚/耦合，并概述最小、安全的拆分
- 如果重构非平凡，提出增量计划而非大规模重写

### 3) 移除候选 + 迭代计划

- 加载 `references/removal-plan.md` 获取模板
- 识别未使用、冗余或功能标志关闭的代码
- 区分安全立即删除 vs 带计划延期
- 提供包含具体步骤和检查点（测试/指标）的后续计划

### 4) 安全和可靠性扫描

- 加载 `references/security-checklist.md` 获取覆盖范围
- 检查：
  - XSS、注入（SQL/NoSQL/命令）、SSRF、路径遍历
  - AuthZ/AuthN 缺口、缺少租户检查
  - 秘密泄露或日志/环境/文件中的 API 密钥
  - 速率限制、无界循环、CPU/内存热点
  - 不安全反序列化、弱加密、不安全默认值
  - 竞态条件：并发访问、检查后操作、TOCTOU、缺少锁
- 同时指出可利用性和影响

### 5) 代码质量扫描

- 加载 `references/code-quality-checklist.md` 获取覆盖范围
- 检查：
  - 错误处理：吞掉异常、过于宽泛的 catch、缺少错误处理、异步错误
  - 性能：N+1 查询、热路径中的 CPU 密集操作、缺少缓存、无界内存
  - 边界条件：null/undefined 处理、空集合、数值边界、差一错误
- 标记可能导致静默失败或生产事故的问题

### 6) 输出格式

按以下结构组织审查：

```markdown
## 代码审查摘要

**审查文件数**：X 个文件，Y 行变更
**总体评估**：[APPROVE / REQUEST_CHANGES / COMMENT]

---

## 发现

### P0 - 关键
（无或列表）

### P1 - 高
1. **[文件:行号]** 简短标题
  - 问题描述
  - 建议修复

### P2 - 中
2. （跨部分继续编号）
  - ...

### P3 - 低
...

---

## 移除/迭代计划
（如适用）

## 额外建议
（可选改进，非阻塞）
```

内联注释：对此格式用于特定文件的发现：

```
::code-comment{file="path/to/file.ts" line="42" severity="P1"}
问题描述和建议修复。
::
```

清洁审查：如果未发现问题，明确说明：
- 检查了什么
- 未覆盖的区域（例如，"未验证数据库迁移"）
- 残余风险或建议的后续测试

### 7) 下一步确认

展示发现后，询问用户如何继续：

```markdown
---

## 下一步

我发现 X 个问题（P0: _、P1: _、P2: _、P3: _）。

**您希望如何继续？**

1. **全部修复** - 我将实施所有建议的修复
2. **仅修复 P0/P1** - 解决关键和高优先级问题
3. **修复特定项** - 告诉我要修复哪些问题
4. **不做更改** - 审查完成，无需实施
```

请选择一个选项或提供具体指示。

重要：在用户明确确认之前不要实施任何变更。这是先审查工作流。

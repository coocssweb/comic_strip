# 系统化debugging

## 概述

随机修复浪费时间，还会引入新 bug。快速补丁只会掩盖根本问题。

**核心原则：** 必须先找到根因，再尝试修复。只治症状就是失败。

**违反流程的字面要求，就是违反Debugging的精神。**

## 铁律

```
没有完成根因调查，禁止提出修复方案
```

如果没有完成阶段一，不允许提出任何修复。

## 适用场景

适用于任何技术问题：
- 测试失败
- 生产环境 bug
- 非预期行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况必须使用：**
- 时间紧迫（越急越容易瞎猜）
- 觉得「改一下就好了」
- 已经试了好几个修复方案
- 上一个修复没生效
- 对问题理解不充分

**以下情况也不能跳过：**
- 问题看起来很简单（简单 bug 也有根因）
- 赶时间（系统化Debugging比瞎试更快）

## 四个阶段

必须按顺序完成每个阶段，才能进入下一阶段。

### 阶段一：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 错误信息通常包含解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠触发？
   - 具体步骤是什么？
   - 每次都能复现吗？
   - 无法复现 → 收集更多数据，不要猜

3. **检查近期变更**
   - 什么改动可能导致了这个问题？
   - Git diff、最近的提交
   - 新依赖、配置变更
   - 环境差异

4. **多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断埋点：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每一层的状态

   运行一次收集证据，定位断裂点
   然后分析证据，识别故障组件
   然后针对该组件深入调查
   ```

5. **追踪数据流**

   **当错误发生在调用栈深处时：**
   - 错误值从哪里来？
   - 谁用错误值调用了这里？
   - 持续向上追踪直到找到源头
   - 在源头修复，不在症状处修复

### 阶段二：模式分析

**修复之前先找到规律：**

1. **找到可用的参考示例**
   - 在同一代码库中找到类似的正常代码
   - 什么类似的功能是正常工作的？

2. **对照参考实现**
   - 如果在实现某个模式，完整阅读参考实现
   - 不要略读，逐行阅读
   - 完全理解模式后再应用

3. **识别差异**
   - 正常代码和异常代码之间有什么不同？
   - 列出所有差异，无论多小
   - 不要假设「这个不可能有影响」

4. **理解依赖关系**
   - 这个功能依赖哪些其他组件？
   - 需要什么配置、环境？
   - 它做了哪些假设？

### 阶段三：假设与验证

**科学方法：**

1. **提出单一假设**
   - 明确表述：「我认为 X 是根因，因为 Y」
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的改动来验证假设
   - 一次只改一个变量
   - 不要同时修复多个问题

3. **验证后再继续**
   - 生效了？→ 进入阶段四
   - 没生效？→ 提出新假设
   - 不要在失败的修复上叠加更多修复

4. **不确定时**
   - 说「我不理解 X」
   - 不要假装知道
   - 寻求帮助
   - 做更多调研

### 阶段四：实施修复

**修复根因，不是症状：**

1. **创建失败测试用例**
   - 最简复现
   - 尽可能自动化测试
   - 修复前必须有

2. **实施单一修复**
   - 针对已识别的根因
   - 一次只改一处
   - 不要「顺手」改进
   - 不要捆绑重构

3. **验证修复**
   - 测试通过了？
   - 其他测试没被破坏？
   - 问题真的解决了？

4. **修复不生效时**
   - 停下来
   - 数一下：已经试了几次修复？
   - 少于 3 次：回到阶段一，用新信息重新分析
   - **3 次及以上：停下来，质疑架构（见第 5 步）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **3 次以上修复失败：质疑架构**

   **表明架构有问题的信号：**
   - 每次修复都暴露出不同位置的新问题
   - 修复需要「大规模重构」才能实施
   - 每次修复都在其他地方产生新症状

   **停下来质疑根本问题：**
   - 这个模式从根本上是否合理？
   - 是不是因为惯性才坚持用它？
   - 应该重构架构还是继续修补症状？

   **在尝试更多修复之前，先和用户讨论**

## 危险信号 - 立即停下来

如果你发现自己在想：
- 「先快速修一下，以后再查」
- 「试试改 X 看看行不行」
- 「一起改几个地方，跑一下测试」
- 「跳过测试，手动验证一下」
- 「大概是 X 的问题，改了试试」
- 「我不完全理解，但这个可能有用」
- 「文档说要用 X，但我换个方式」
- 没有调查就列出修复方案
- **「再试一次」（已经试了 2 次以上）**

**以上全部意味着：停下来，回到阶段一。**

## 常见借口

| 借口 | 现实 |
|------|------|
| 「问题很简单，不需要流程」 | 简单 bug 也有根因，流程对简单问题很快 |
| 「紧急情况，没时间走流程」 | 系统化Debugging比瞎猜快得多 |
| 「先试这个，再调查」 | 第一次修复定下基调，从一开始就做对 |
| 「确认修复有效后再写测试」 | 没测试的修复不可靠，先写测试才能证明 |
| 「一次改多个省时间」 | 无法隔离哪个生效了，还会引入新 bug |
| 「我看到问题了，直接改」 | 看到症状 ≠ 理解根因 |
| 「再试一次」（已失败 2 次以上） | 3 次以上失败 = 架构问题，不要继续修补 |

## 速查表

| 阶段 | 关键活动 | 完成标准 |
|------|---------|---------|
| **1. 根因调查** | 读错误信息、复现、检查变更、收集证据 | 理解了什么出了问题以及为什么 |
| **2. 模式分析** | 找参考示例、对比 | 识别出差异 |
| **3. 假设验证** | 提出假设、最小化测试 | 假设被确认或提出新假设 |
| **4. 实施修复** | 创建测试、修复、验证 | bug 解决，测试通过 |

## 辅助技术

### 根因追踪

bug 通常在调用栈深处暴露。本能反应是在错误出现的地方修复，但那只是治标。

**沿调用链向上追踪，直到找到原始触发点，然后在源头修复。**

1. 观察症状
2. 找到直接原因
3. 问：谁调用了这里？
4. 继续向上追踪
5. 找到原始触发点

**永远不要只在错误出现的地方修复。** 追溯到原始触发点。

### 纵深防御验证

修复由无效数据导致的 bug 时，在数据经过的每一层都添加验证：

- **第一层：入口验证** - 在 API 边界拒绝明显无效的输入
- **第二层：业务逻辑验证** - 确保数据对当前操作有意义
- **第三层：环境守卫** - 在特定上下文中阻止危险操作
- **第四层：Debugging埋点** - 捕获上下文用于事后分析

### 基于条件的等待

用条件轮询替代固定延时：

```javascript
// ❌ 猜测时间
await new Promise(r => setTimeout(r, 50));

// ✅ 等待条件满足
await waitFor(() => getResult() !== undefined);
```

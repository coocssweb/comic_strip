# 头脑风暴：从想法到设计

通过自然的协作对话，将模糊的想法转化为完整的设计方案和规格说明。

先了解当前项目上下文，然后逐个提问来细化想法。理解清楚要做什么之后，呈现设计方案并获得用户确认。

<HARD-GATE>
在呈现设计方案并获得用户批准之前，禁止调用任何实现类 skill、编写任何代码、搭建任何项目结构，或采取任何实现行动。无论项目看起来多简单，都必须走这个流程。
</HARD-GATE>

## 反模式："这个太简单了，不需要设计"

每个项目都要走这个流程。一个 todo 列表、一个工具函数、一个配置变更 — 全都要。"简单"的项目恰恰是未经审视的假设造成最多返工的地方。设计可以很短（真正简单的项目几句话就够），但你必须呈现出来并获得确认。

## 检查清单

你必须按顺序完成以下每一项：

1. **探索项目上下文** — 查看文件、文档、最近的提交
2. **提供可视化辅助**（如果话题涉及视觉相关问题）— 这是一条独立消息，不与澄清问题合并。详见下方"可视化辅助"章节。
3. **逐个提出澄清问题** — 一次一个，理解目的/约束/成功标准
4. **提出 2-3 种方案** — 附带权衡分析和你的推荐
5. **呈现设计方案** — 按复杂度分段呈现，每段之后确认用户是否认可
6. **编写设计文档** — 保存到 `docs/specs/YYYY-MM-DD-<主题>-design.md` 并提交
7. **规格自审** — 快速检查占位符、矛盾、歧义、范围（详见下方）
8. **用户审阅规格** — 请用户在继续之前审阅规格文件
9. **过渡到实现** — 创建实现计划

## 流程图

```

    │否                              ↓
    ↓                          逐个提出澄清问题

    ↓

    ↓

    │是
    ↓
```

**终态是创建实现计划。** 不要跳过计划直接调用任何实现类 skill。头脑风暴之后的唯一下一步是制定实现计划。

## 具体流程

### 理解想法

- 先查看当前项目状态（文件、文档、最近提交）
- 在深入提问之前，先评估范围：如果需求描述了多个独立子系统（比如"做一个包含聊天、文件存储、计费和分析的平台"），立即指出这一点。不要花时间细化一个需要先拆分的项目。
- 如果项目太大，无法用一份规格覆盖，帮用户拆分为子项目：有哪些独立模块、它们之间的关系、应该按什么顺序构建？然后对第一个子项目走正常的设计流程。每个子项目都有自己的 规格 → 计划 → 实现 周期。
- 对于范围合适的项目，逐个提问来细化想法
- 尽量用选择题，开放式问题也可以
- 每条消息只问一个问题 — 如果一个话题需要更多探索，拆成多个问题
- 聚焦于理解：目的、约束、成功标准

### 探索方案

- 提出 2-3 种不同方案，附带权衡分析
- 以对话方式呈现选项，给出你的推荐和理由
- 先说推荐方案，再解释为什么

### 呈现设计

- 当你认为已经理解了要做什么，呈现设计方案
- 每个章节的篇幅与其复杂度匹配：简单的几句话，复杂的 200-300 字
- 每个章节之后询问是否正确
- 覆盖：架构、组件、数据流、错误处理、测试
- 如果有不清楚的地方，随时回头澄清

### 设计原则：隔离与清晰

- 将系统拆分为更小的单元，每个单元有一个明确的职责，通过定义良好的接口通信，可以独立理解和测试
- 对于每个单元，你应该能回答：它做什么、怎么用、依赖什么？
- 别人能否不看内部实现就理解一个单元的功能？能否修改内部实现而不影响调用方？如果不能，边界需要调整。
- 更小、边界清晰的单元也更容易处理 — 能在上下文中完整理解的代码，推理更准确，编辑更可靠。当一个文件变得很大时，通常意味着它承担了太多职责。

### 在现有代码库中工作

- 在提出变更之前先探索现有结构，遵循已有模式。
- 如果现有代码存在影响当前工作的问题（比如文件过大、边界不清、职责纠缠），将针对性的改进纳入设计 — 就像一个好的开发者在工作中顺手改进代码一样。
- 不要提出无关的重构。聚焦于服务当前目标的改动。

## 设计完成后

### 文档

- 将验证通过的设计（规格）写入 `docs/specs/YYYY-MM-DD-<主题>-design.md`
  - （用户对规格位置的偏好优先于此默认路径）
- 如果有 elements-of-style:writing-clearly-and-concisely skill 可用，使用它优化文档表达
- 将设计文档提交到 git

### 规格自审

编写规格文档后，以全新视角审视：

1. **占位符扫描：** 是否有 "TBD"、"TODO"、未完成的章节或模糊的需求？修复它们。
2. **内部一致性：** 各章节之间是否有矛盾？架构是否与功能描述匹配？
3. **范围检查：** 是否足够聚焦，可以用一份实现计划覆盖？还是需要拆分？
4. **歧义检查：** 是否有需求可以被两种方式理解？如果有，选定一种并明确写出。

就地修复所有问题。无需重新审阅 — 直接修复然后继续。

审阅检查项的详细标准见：`spec-document-reviewer-prompt.md`

### 用户审阅关卡

规格自审通过后，请用户审阅：

> "规格已编写并提交到 `<路径>`。请审阅一下，如果需要修改请告诉我，确认后我们开始制定实现计划。"

等待用户回复。如果用户要求修改，修改后重新执行规格自审。只有用户确认后才继续。

### 实现

- 创建详细的实现计划（可使用 spec 功能）
- 不要跳过计划直接写代码

## 核心原则

- **一次一个问题** — 不要用多个问题轰炸用户
- **优先选择题** — 比开放式问题更容易回答
- **严格 YAGNI** — 从所有设计中移除不必要的功能
- **探索替代方案** — 在确定之前始终提出 2-3 种方案
- **增量验证** — 呈现设计，获得确认后再继续
- **保持灵活** — 有不清楚的地方随时回头澄清

## 可视化辅助

当头脑风暴涉及视觉相关问题（UI 布局、架构图、设计对比等）时，可以借助浏览器展示模型图、图表和视觉选项。它是一个工具，而非模式。用户同意使用可视化辅助，意味着它可用于受益于视觉展示的问题；并不意味着每个问题都要通过浏览器。

**提供辅助的时机：** 当你预判后续问题会涉及视觉内容（模型图、布局、图表），征求一次同意：

> "接下来讨论的内容可能用视觉方式展示更直观。我可以在浏览器中展示模型图、图表、对比方案等。这个功能还比较新，可能会消耗较多 token。需要试试吗？（需要打开一个本地 URL）"

**这必须是一条独立消息。** 不要与澄清问题、上下文总结或其他内容混在一起。等待用户回复后再继续。如果用户拒绝，继续纯文本的头脑风暴。

**逐个问题决策：** 即使用户同意了，也要对每个问题单独判断是用浏览器还是终端。判断标准：**用户看到它是否比读到它更容易理解？**

- **用浏览器** 展示本身就是视觉的内容 — 模型图、线框图、布局对比、架构图、并排设计对比
- **用终端** 处理文本内容 — 需求问题、概念选择、权衡列表、A/B/C 文字选项、范围决策

关于 UI 话题的问题不一定是视觉问题。"这个场景下个性化是什么意思？"是概念问题 — 用终端。"哪种向导布局更好？"是视觉问题 — 用浏览器。

如果用户同意使用可视化辅助，在继续之前先阅读详细指南：
`visual-companion.md`

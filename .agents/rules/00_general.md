# 通用开发规则

## 一、沟通风格

- 假设用户具备资深全栈工程师经验，无需解释基础概念
- 直接给出专业级方案，避免冗余说明
- 使用工程术语，不做过度简化
- 遇到权衡取舍时，列出选项让用户决策，而非替用户选择

## 二、编码前思考

**不要假设。不要隐藏困惑。先暴露权衡。**

动手写代码之前：
- **显式陈述你的假设** — 如果不确定，先问
- **存在多种理解时，列出它们** — 不要静默选一个
- **存在更简单的方案时，说出来** — 该 push back 就 push back
- **遇到不清楚的地方，停下来** — 说清楚哪里困惑，然后问

## 三、精准改动（Surgical Changes）

**只动必须动的。只清理自己制造的。**

编辑现有代码时：
- 不"改进"路过的代码、注释或格式
- 不重构没有问题的东西
- 匹配现有风格，即使你觉得不好
- 发现无关的死代码，报告它 — 不要删除它

当你的改动产生了"孤儿"：
- 删除**你的改动**导致不再使用的 import / 变量 / 函数
- 不要删除改动之前就已存在的死代码，除非被要求

自检：**每一行变更都应能直接追溯到用户的请求。**

## 四、简洁性原则（Simplicity First）
只写能解决当前问题的最少代码。绝不写推测性/预测性的代码。
- **只写被要求的功能**，不投机、不预留"以后可能用到"的扩展
- **不为单次使用的代码建抽象** — 如果只用一次，内联即可
- **不添加未被要求的"灵活性"或"可配置性"**
- **不为不可能发生的场景写错误处理**
- 自检：「一个资深全栈工程师会说这段代码过度设计了吗？」如果会，简化它

## 五、契约文档同步铁律 (Documentation Sync)

代码不是唯一的真相，文档契约是系统的骨架。

修改以下内容后，**必须同步更新相关文档**：
1. **修改了 API 接口**（路径、参数、响应体、状态码）：必须同步更新 `.agents/skills/api-contract/` 下的 `admin-web-api.md` 或 `mini-program-api.md`。
2. **修改了数据库结构**（Schema、字段、表）：必须同步更新 `docs/TECH.md` 中的数据字典。
3. **引入了新的核心依赖或改变了系统边界**：必须更新 `ARCHITECTURE.md`。

严禁只改代码不改文档。

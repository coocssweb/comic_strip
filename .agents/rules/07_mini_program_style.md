# 小程序样式规范

适用于 `mini-program/miniprogram` 下所有 WXML 与 WXSS 的新增、修改和审查。目标是让样式以语义类和设计 token 为边界，页面与组件能够稳定复用，而不是把视觉规则散落在模板属性中。

## 一、执行顺序

新增或修改界面时，按以下顺序处理：

1. 检索同类页面、组件和 `components/` 中已有样式能力，优先复用。
2. 先用语义类组织 WXML 结构，再在同目录 WXSS 中实现样式。
3. 先使用 `app.wxss` 已有设计 token；确有新增设计语义时，再补充 token。
4. 只有运行时连续变化且无法用类名表达的值，才使用受控的 CSS 自定义属性。

完成条件：新增或修改的每个视觉属性都能追溯到语义类、设计 token，或本规范定义的动态样式例外。

## 二、样式的唯一入口

### WXML 使用语义类

- 默认通过 `class` 绑定样式，页面、组件和可见元素不得依赖静态 `style` 属性。
- 类名描述业务或结构角色，不描述临时外观；使用 `comic-card`、`comic-card__cover`，不使用 `red-text`、`mt-16`、`big-title`。
- 固定状态和有限变体通过修饰类表达，例如：

```xml
<view class="comic-card {{isSelected ? 'comic-card--selected' : ''}}">
  <text class="comic-card__title">{{title}}</text>
</view>
```

- 不从接口数据、页面数据或字符串拼接中直接注入任意类名；可选类名必须由当前 WXML 中明确列出的状态分支产生。
- `hidden`、`wx:if`、`wx:for` 等结构属性只控制结构和状态；视觉效果仍由类名与 WXSS 管理。

### 禁止静态内联样式

- 禁止在 WXML 中写 `style="color: ..."`、`style="margin: ..."`、`style="width: ..."` 等静态样式，也禁止在 JS 中拼接整段 CSS 字符串。
- 禁止以 `data-*`、通用间距类或页面级全局选择器充当样式逃生通道。
- 需要调整外部位置时，由父页面的包装元素负责，例如 `page-library__comic-list`；组件根节点默认不携带外边距。

## 三、动态样式例外

运行时测量值、百分比进度和安全区偏移等连续变化的数值，可以通过单个 CSS 自定义属性传入。使用前必须确认固定状态类无法表达该需求。

```xml
<view class="reading-progress" style="--reading-progress: {{progress}}%">
  <view class="reading-progress__bar" />
</view>
```

```css
.reading-progress__bar {
  width: var(--reading-progress);
}
```

动态样式必须同时满足：

- `style` 中只允许声明 CSS 自定义属性，禁止直接声明 `width`、`color`、`margin` 等视觉属性。
- 变量名以组件或页面块名开头，例如 `--reading-progress`，避免跨组件冲突。
- 传入值必须是页面自身计算出的受控数值；在 JS 中完成类型转换和范围约束，不直接使用接口返回的原始字符串。
- 单一变量只表达一个运行时维度；出现多个独立视觉维度时，重新评估是否应拆为状态类或组件参数。

## 四、设计 token

### 统一来源

- 颜色、字号、字重、行高、间距、圆角、阴影、层级和动效时长等设计值使用 token，统一维护在 `miniprogram/app.wxss`。
- 新 token 以设计语义命名，例如 `--color-text-muted`、`--spacing-section`、`--radius-card`；禁止按页面、组件或具体色值命名。
- token 只表达稳定的设计语言，不为单个页面的偶发几何尺寸创建 token。与图片比例、进度值或业务容量直接绑定的尺寸可在局部使用，并在相邻注释中说明约束来源。
- 新 token 只有在当前值具有明确设计语义或预计被多个界面共享时才添加；单一组件内部的私有尺寸保留在该组件 WXSS 中。

### 值的使用规则

- 所有静态颜色必须使用颜色 token；反色文字、遮罩、禁用态等语义也要有对应 token。
- 常规字体大小、间距、圆角、阴影和动效时长优先使用 token，不能在业务 WXSS 中重复散落同类字面量。
- `0`、`100%`、`auto`、`flex`、`1rpx` 边框宽度等结构性值可直接使用；系统导航栏等平台配置尺寸按微信要求处理。
- 页面和组件使用 token 时只引用，不在各自 WXSS 中重新定义同名 token。主题或全局视觉调整只修改 `app.wxss` 的权威定义。

## 五、类名与选择器

### 块命名

- 页面根块使用 `page-` 前缀加页面目录语义，例如 `page-library`、`page-reader`。
- 组件根块使用组件目录名的 kebab-case，例如 `comic-card`、`reading-progress`。
- 块内元素使用 `块名__元素`，状态或变体使用 `块名--状态`，例如 `comic-card__cover`、`comic-card--selected`。
- 类名只使用小写英文、数字和连字符；不得使用拼音、无语义缩写或通用名如 `container`、`title`、`item`、`btn`。

### 选择器边界

- 业务样式以单个类选择器为主；全局 reset 可使用元素选择器。
- 避免依赖标签层级、页面路径或组件宿主的选择器。确有必要的内部关系选择器最多两层，且两侧都必须是当前块的语义类。
- 禁止 ID 选择器、通配符选择器、`!important` 和跨页面覆盖。第三方组件无法避免 `!important` 时，需在相邻注释中写明组件名与原因。
- 页面样式不得穿透或假定组件内部结构；组件只在自己的 WXSS 中声明内部元素样式。

## 六、布局与组件边界

- 页面负责页面区域、列表排布、组件之间的间距和安全区；组件负责自身内部结构、内容布局和视觉状态。
- 组件根节点不默认设置外边距，不依赖父页面的标签选择器，也不暴露未约定的内部类名。
- 同一种可交互视觉结构在第二个真实使用点出现时，提炼为 `components/` 下的公共组件；仅样式相似但数据和交互模型不同的内容不强行合并。
- 自定义组件 JSON 默认设置 `"styleIsolation": "isolated"`。确需改变隔离策略时，必须在 JSON 相邻说明中写明被复用的样式边界和原因。
- `<image>` 的尺寸、圆角和裁切通过语义类控制，且必须显式设置符合内容语义的 `mode`。

## 七、WXSS 书写顺序与适配

- 一个页面或组件的 WXSS 按“根块 → 元素 → 状态/变体 → 媒体或平台兼容”的顺序组织；相邻规则保持同一块名，方便审查。
- 单条规则内按“布局 → 盒模型 → 尺寸 → 排版 → 视觉 → 动效”排序。属性不超过需要的范围，不为未来场景预留样式。
- 布局优先使用 `flex`、百分比、`rpx` 与 `calc()`；业务界面不得硬编码 `px`。安全区、系统导航栏和平台 API 返回的物理尺寸属于例外。
- 触控目标的可点击区域由容器尺寸或 padding 保证，不通过透明占位元素、负 margin 或绝对定位补偿点击范围。
- 绝对定位仅用于覆盖层、角标等明确的叠放关系；叠放关系使用语义化层级 token，不能用递增的任意 `z-index` 解决。

## 八、审查清单

提交样式前逐项确认：

- 每个 WXML 的 `style` 属性都符合“动态样式例外”；没有静态样式和拼接 CSS 字符串。
- 新增视觉值已使用已有 token，或已在 `app.wxss` 补充一次权威定义。
- 类名符合块、元素、状态命名，且没有 `container`、`item`、`btn` 等泛名。
- 页面没有覆盖组件内部，组件没有决定外部排版；`styleIsolation` 与实际边界一致。
- WXSS 中没有 `!important`、ID、通配符和不必要的多层选择器。
- 在微信开发者工具中检查过至少一个窄屏与一个宽屏模拟器，确认没有裁切、重叠或可点击区域错位。

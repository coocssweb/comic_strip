# 通用工程规范

本规范适用于 `backend`、`mini-program`、`admin-web` 的所有代码与文档变更。它只约束与框架无关的基础工程质量；React 组件拆分、Hooks、样式规则分别放在对应前端规范中。

## 一、命名规则

- 变量、函数、类、接口、类型、文件名中的技术标识使用英文，禁止拼音命名。
- 命名必须表达业务语义，禁止 `data`、`item`、`list`、`temp`、`obj` 等无上下文泛名长期存在。
- 单字母变量只允许用于极短作用域内的数学或数组遍历场景；业务逻辑中禁止使用。
- 布尔值使用 `is`、`has`、`can`、`should`、`allow` 等前缀表达判断语义。
- 事件处理函数使用 `handle` 前缀；传给子组件的事件回调用 `on` 前缀。
- 常量使用全大写下划线或清晰的领域名；魔法字符串、魔法数字不得散落在业务流程中。
- 不允许为了“看起来通用”创建含义模糊的 `helper`、`common`、`utils2`、`newUtils`。

## 二、注释规则

- 注释必须解释“为什么”和“业务约束”，不要复述代码做了什么。
- 以下位置必须写中文注释：非直观业务规则、API 字段映射、状态流转、权限判断、数据兼容逻辑、临时方案及移除条件。
- 禁止编号式流水账注释，例如“1. 获取数据”“2. 提交保存”，除非编号本身对应真实业务步骤。
- 禁止保留误导性注释；修改逻辑时必须同步更新相关注释。
- 临时兼容代码必须标明触发条件、移除条件和关联上下文。

## 三、类型与数据结构

- 新增复杂对象时，必须明确数据结构来源：后端响应、表单状态、视图模型、缓存对象或组件 props。
- 前端页面不得直接把后端原始响应层层传递到深层组件；跨组件传递前应归一化为视图模型。
- 后端响应字段、请求字段和枚举值不得在页面中臆造；必须先对照对应 API 契约。
- 可为空字段必须显式处理 `null`、`undefined`、空字符串和空数组的语义差异。
- 同一业务事实只能有一个权威来源；后端响应、视图模型、缓存对象之间不得无约束复制字段，必须通过明确的映射函数转换。
- 对外暴露的函数参数超过 3 个时，应考虑参数对象，并为参数对象命名表达业务含义。

## 四、类型契约

- 导出函数、导出类、公共 hook、跨文件复用工具函数必须声明参数和返回值契约。
- React 导出组件、具名子组件、跨文件复用组件必须声明 props 契约；禁止新增无类型 props 的可复用组件。
- TypeScript 文件使用 `interface Props` 或 `type Props` 描述 props，不允许把 props 标成宽泛的 `any` 或无字段约束对象。
- JavaScript/JSX 文件使用 `PropTypes` 描述 props；如果当前子项目尚未引入 `prop-types`，新增可复用组件前必须先补依赖、迁移到 TS，或在任务中明确说明豁免原因。
- 仅在同一 render 分支内使用、没有独立组件名、没有跨文件传递的内联 JSX 片段，可以不单独声明 props 契约。
- 类型契约必须覆盖必填性、可空性、枚举值、回调参数和集合元素结构。

## 五、错误处理

- 不允许空 `catch`，也不允许只 `console.log` 后静默失败。
- 用户可见错误必须使用中文，并避免暴露内部堆栈、数据库字段、接口路径等细节。
- 技术日志必须包含足够排查上下文，例如业务 ID、操作名、关键参数摘要；不得记录令牌、密码、身份证、手机号等敏感明文。
- API 业务失败按项目响应 `code` 判断，不只依赖 HTTP 状态。
- 可恢复错误要给出恢复路径，例如重试、重新登录、刷新、重新选择文件。
- 不可恢复错误要在合适边界中断流程，避免后续逻辑基于错误状态继续执行。

## 六、函数与模块边界

- 函数只承担一个明确意图：数据转换、校验、请求、渲染辅助、状态流转不要混在一个函数里。
- 纯函数不得读写外部状态、调用网络、操作 DOM 或写日志。
- 业务函数不得依赖偶然的组件局部变量；需要依赖时通过参数显式传入。
- 公共工具函数必须有第二个真实调用场景；单次使用优先内联或放在当前模块私有区域。
- 不允许为了预测未来需求创建过度抽象、过宽参数或可配置项。

## 七、常量与配置

- 颜色、间距、分页大小、状态枚举、上传类型、错误码等值不得散落重复定义。
- 与 API 契约相关的常量必须与契约文档保持同步。
- 与 UI 设计系统相关的常量应进入前端样式 token 或局部配置，不要写在业务流程中。
- 只在当前文件使用且语义明确的常量可以放在文件顶部；跨文件复用时再提升到领域常量文件。

## 八、代码生成与审查要求

- AI 生成代码必须经过人工级审查，重点检查：重复造轮子、隐式状态、硬编码、无意义注释、依赖缺失、错误吞掉。
- 修改已有代码时只处理与当前任务直接相关的问题；发现无关问题应记录或反馈，不顺手扩大改动。
- 新增规范后，相关 `AGENT.md` 必须同步引用，避免规则只存在于文件中而不进入工作流。

# 不采用 COS 事件通知回调补偿链路

#6 原始决策纳入了 COS 事件通知 → 云函数 → 后端 Webhook 异步回调作为图片绑定的「漏单补偿」。经压力测试后决定裁掉整条链路，理由如下：

- 两阶段绑定的核心保障已在绑定确认时由后端直接校验 COS 对象是否存在，云函数回调不承担不可替代的一致性责任
- 云函数回调链的每一跳（COS → 云函数 → 后端）都可能丢失，且修复代价（云函数部署、鉴权、重试逻辑、监控）远超其实际兜住的场景（管理员上传后忘记点绑定）
- 项目已接受未绑定图片永久占用 COS 存储、由管理员按需手动清理的运维模型——云函数回调加速的那个「提醒」不改变最终结果

**考虑过的替代方案**：保留云函数回调但降级为仅记录日志。被拒绝，因为仅记录日志不产生任何业务价值，仍然需要维护云函数链路。

**后果**：图片绑定的唯一确认路径是管理端主动提交绑定请求 + 后端校验 COS 对象。如果未来管理员频繁忘记绑定，可以选择加一个定时扫描 COS 中无对应绑定的对象、生成清单供管理员确认或清理的轻量方案——但那是运维工具而非核心链路，不在此 ADR 范围。

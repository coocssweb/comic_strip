## Problem Statement

当前四格漫画小程序处于绿地状态。游客可以浏览公开内容，但无法执行需要身份的互动（点赞、收藏、评论），因为系统尚不支持从微信登录中派生业务用户身份、建立独立业务会话、以及管理用户的隐私同意和账号生命周期。唯一管理员认证体系已完成，但用户侧的身份基础设施完全空白。

## Solution

为小程序提供一套完整的用户身份与业务会话系统：微信小程序登录换取 `openid` → 首次互动时触发隐私同意 → 自动建档注册为「注册用户」→ 签发独立的「业务会话」。业务会话与微信侧会话完全解耦，支持多设备同时在线、活跃自动续期，失效时通过微信身份静默恢复。同时提供退出登录、账号注销（不设冷静期）、重新注册（全新业务身份，不认领历史数据）的完整生命周期。

## User Stories

1. 作为游客，在小程序中浏览漫画时可以看到公开内容和排行榜，不需要登录或注册，系统也不记录我的任何个人数据。
2. 作为游客，首次点击点赞/收藏/发表评论按钮时，看到《隐私协议》弹窗，明确告知我的个人信息将如何被处理，我需要明确同意才能继续。
3. 作为游客，阅读《隐私协议》后点击「拒绝」，则保持游客身份继续浏览，点赞/收藏/发表评论操作被取消，不会再次自动弹出隐私同意弹窗（除非我主动触发下一个需要身份的互动）。
4. 作为游客，阅读《隐私协议》后点击「同意」，系统自动为我创建业务身份（注册用户），我无需填写任何表单、选择用户名或设置密码，注册过程完全无感。
5. 作为注册用户，在我的业务会话有效期内，可以正常点赞、收藏漫画和发表评论，无需每次操作都重新验证身份。
6. 作为注册用户，我可以在手机和平板上同时登录小程序，两个设备的业务会话互不影响。
7. 作为注册用户，当我持续使用小程序时，我的业务会话会自动续期——我不需要担心在阅读漫画过程中会话过期。
8. 作为注册用户，当我长时间不使用小程序后再次打开时，即使业务会话已过期，系统也会通过我的微信身份静默恢复会话，我无需手动重新登录。
9. 作为注册用户，我可以主动退出登录——退出只终止当前设备的业务会话，不影响其他设备。
10. 作为注册用户，我可以在设置中选择注销账号——注销后我的点赞和收藏被清空，历史评论以「已注销用户」匿名保留，我的业务身份被永久终止。
11. 作为注册用户，在注销账号前，系统要求我重新验证微信身份并二次确认，防止误操作。注销即时生效且不可撤销。
12. 作为已注销用户，我可以使用同一微信身份重新注册小程序——重新注册会创建全新的业务身份，我无法认领之前注销账号的任何数据。
13. 作为唯一管理员，在管理端可以查看注册用户列表，并对违规用户执行账号禁用——禁用立即终止该用户的所有业务会话，使其退回游客只读状态。
14. 作为唯一管理员，可以对已禁用的注册用户执行账号恢复——恢复后用户可重新登录，历史互动（点赞、收藏、评论）归原账号管理。
15. 作为被禁用的注册用户，我的历史评论仍以原用户名展示给公众，但我在禁用期间不能发表新评论或执行任何互动。
16. 作为游客，当我清空小程序的本地数据/授权状态后，系统根据我的微信 openid 识别我之前是否已完成隐私同意——如果已完成且账号未被禁用/注销，直接恢复业务会话，不重复弹出隐私同意弹窗。

## Implementation Decisions

### 微信登录流程

采用微信小程序标准登录流程：`wx.login()` 获取临时 code → 后端使用 code 向微信服务器换取 `openid` 和 `session_key`。`session_key` 仅用于本次请求内的微信数据解密（如获取用户信息），严禁直接作为业务会话 Token 或跨请求复用。

### 游客访问

- 游客请求不携带任何身份凭据，后端判定为游客后仅返回公开数据。
- 游客访问期间不创建任何数据库记录，不跟踪行为，不形成可合并的本地权益。
- 首次访问时，后端不主动触发注册流程。

### 隐私同意与自动注册

- 当游客执行首次需要身份的互动时，后端返回特定的业务错误码（如 `WECHAT_PRIVACY_CONSENT_REQUIRED`），前端收到后弹出《隐私协议》弹窗。
- 用户同意后，前端调用注册接口。后端在该请求中完成：微信 code → openid 交换 → `registered_users` 文档创建 → 隐私同意状态记录 → 业务会话签发。
- 用户拒绝后，前端锁定互动入口（不向后端发起请求），保持游客状态。
- 隐私同意状态独立存储，清空后可重新确认，但清空不改变注册状态。

### 业务会话

- 业务会话是独立的 Bearer Token（JWT 格式），通过 HTTP `Authorization` 头传递。
- 会话签发时在 `user_sessions` 集合中建立记录，包含：会话 ID、用户 ID、创建时间、最近活跃时间、空闲超时、绝对超时。
- 每次携带有效会话的请求到达时，更新 `lastSeenAt` 并重新计算 `idleExpiresAt`（`min(lastSeenAt + 30 分钟, absoluteExpiresAt)`）。
- 绝对超时从签发时刻起算 7 天。空闲超时 30 分钟。
- 同一用户可同时持有多个有效业务会话（多设备）。
- 会话失效后，前端使用微信 code 重新调用会话恢复接口——后端验证 openid 匹配、账号未被禁用/注销后签发新会话，不弹出隐私同意弹窗。

### 账号注销

- 前端调用注销接口时，需携带最新的微信 code 用于二次身份校验。
- 后端验证通过后执行原子操作：删除 `user_sessions` 中该用户的所有会话 → 物理删除该用户的 `comic_likes` 和 `comic_favorites` 记录 → 原子 `$inc` 扣减对应漫画的 `likeCount`/`favoriteCount` → 将该用户的所有可见/已隐藏评论的 `userId` 设为 `null` 并标记 `isAnonymous: true` → 标记 `registered_users` 文档为已注销。
- 注销即时生效，不可撤销，不设冷静期。

### 账号禁用与恢复

- 管理员通过管理端 API 对 `registered_users` 文档设置 `disabled: true`。
- 禁用操作同时撤销该用户的所有业务会话（批量标记为 revoked）。
- 禁用不修改历史点赞、收藏、评论数据——评论保留原 `userId` 引用和原作者用户名展示。
- 恢复时清除 `disabled` 标记，用户可重新通过微信登录恢复业务会话。

### 重新注册

- 已注销用户使用同一微信身份再次触发注册时，系统创建全新的 `registered_users` 文档（新 `_id`），微信身份绑定到新文档。
- 历史数据不可认领——这是全新的业务身份。

### 数据模型

新增两个 MongoDB 集合：

**`registered_users`**
| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `_id` | 字符串 (UUID v4) | 必须 |
| `wechatOpenId` | 字符串 | 必须，唯一索引 |
| `privacyConsented` | 布尔 | 必须，默认 false |
| `privacyConsentedAt` | Date | 可选 |
| `disabled` | 布尔 | 必须，默认 false |
| `disabledAt` | Date | 可选 |
| `deactivated` | 布尔 | 必须，默认 false（注销标记） |
| `deactivatedAt` | Date | 可选 |
| `createdAt` | Date | 必须 |
| `updatedAt` | Date | 必须 |

**`user_sessions`**
| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `_id` | 字符串 (UUID v4 / JWT jti) | 必须 |
| `userId` | 字符串 | 必须，索引 |
| `createdAt` | Date | 必须 |
| `lastSeenAt` | Date | 必须 |
| `idleExpiresAt` | Date | 必须，TTL 索引 |
| `absoluteExpiresAt` | Date | 必须 |

### API 契约

所有小程序接口统一前缀 `/api/v1`。成功响应 `{ success: true, data: {...} }`，失败响应 `{ success: false, code: ..., message: ... }`。

| 方法 | 路径 | 鉴权 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/v1/auth/wechat-login` | 无 | 微信 code 换取 openid，若已注册则返回业务会话；若隐私未同意返回 `WECHAT_PRIVACY_CONSENT_REQUIRED` |
| POST | `/api/v1/auth/register` | 无 | 完成隐私同意 + 注册 + 签发业务会话（需携带微信 code） |
| GET | `/api/v1/auth/session` | 用户 | 恢复/刷新当前业务会话，返回最新会话状态 |
| DELETE | `/api/v1/auth/session` | 用户 | 退出登录，仅终止当前设备业务会话 |
| DELETE | `/api/v1/auth/account` | 用户 | 账号注销（需携带最新微信 code 二次确认） |

### 错误码

| 错误码 | HTTP 状态 | 语义 |
| --- | --- | --- |
| `WECHAT_PRIVACY_CONSENT_REQUIRED` | 403 | 游客已通过微信身份识别但未完成隐私同意 |
| `WECHAT_CODE_INVALID` | 400 | 微信临时 code 无效或已过期 |
| `WECHAT_IDENTITY_MISMATCH` | 403 | 请求中的微信身份与当前会话不匹配 |
| `USER_DISABLED` | 403 | 账号已被管理员禁用 |
| `USER_DEACTIVATED` | 403 | 账号已被注销 |
| `USER_ALREADY_REGISTERED` | 409 | 该微信身份已注册（重复注册请求） |
| `REGISTRATION_FAILED` | 500 | 微信身份验证成功但注册写入失败 |

## Testing Decisions

### 测试分层

1. **领域单测** —— 测试 `auth.service.js` 中的纯业务逻辑：注册状态流转（游客 → 隐私同意判断 → 注册 → 会话签发）、会话续期逻辑（空闲超时/绝对超时计算）、注销后数据清理规则。使用内存中的 stub 替代微信 API 和 MongoDB。
2. **MongoDB 集成测试** —— 使用真实 MongoDB（测试库）验证 `registered_users` 的 `wechatOpenId` 唯一索引、`user_sessions` 的 TTL 索引行为、并发注册幂等性。
3. **API 契约测试** —— 对 Koa 应用做黑盒 HTTP 测试，验证 5 个接口的正确状态码、错误码和响应体结构。微信 code 交换使用 mock。
4. **微信接口替身测试** —— 对 `wx.login()` code 交换逻辑使用可控的 mock 服务器，验证正常/过期/无效 code 三种场景。

### 测试原则

- 只测试外部可观察行为（HTTP 状态码、响应体结构、错误码），不测试内部实现细节。
- 优先复用现有测试基础设施：`backend/tests/helpers/test-db.js` 作为 MongoDB 连接管理基类。
- 已有测试文件 `auth.test.js` 覆盖 admin 认证，新增的用户认证测试文件独立命名（如 `user-auth.test.js`）避免混淆。

## Out of Scope

- 小程序端的前端实现（《隐私协议》弹窗 UI、登录流程 UX）—— 属于 mini-program 端 Spec
- 管理端的用户管理界面（用户列表、禁用/恢复操作）—— 属于 admin-web 端 Spec
- 微信 `wx.getUserInfo` / `wx.getUserProfile` 获取用户昵称头像 —— MVP 不收集用户个人信息
- 微信 `unionid` —— MVP 仅使用 `openid`，不涉及多应用统一账号
- 用户关注、私信、通知中心
- 管理员注销用户账号（管理员不能代用户注销）

## Further Notes

- 微信 `session_key` 由微信服务器维护其生命周期，后端仅在使用 code 交换时接收，不存储 `session_key`。
- 业务会话 JWT 的密钥独立于管理会话 JWT 密钥。
- 获取 token 之前，前端必须锁定所有需要身份的互动入口（点赞、收藏、评论按钮），避免并发竞态。
- 后端需要维护 `qcloud-cos-sts` 依赖已就绪，STS 签发将在后续「内容与图片资产」Spec 中实现，不在本 Spec 范围。
- 排行榜重算和 Reconciliation 校准会涉及 `registered_users` 的状态（注销用户不计分），该逻辑在排行榜 Spec 中实现。
